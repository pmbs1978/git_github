Guia Completo de Git e GitHub

Um guia passo a passo para iniciar, gerir e colaborar nos seus projetos.

Introdução

Este tutorial explica de forma clara como criar um repositório Git no seu computador (local) e como conectá-lo a um repositório remoto no GitHub. Dominar este fluxo de trabalho é fundamental para qualquer programador.

A ideia central é que existem dois "locais":

  • Repositório Local: Uma pasta no seu computador onde o Git está a "observar" as alterações nos ficheiros.
  • Repositório Remoto: Um espaço nos servidores do GitHub onde o seu código fica guardado, servindo como backup e permitindo a colaboração.

O Que São o Git e o GitHub?

Muitas vezes, estes dois nomes são usados como sinónimos, mas são duas coisas diferentes que trabalham em conjunto. É crucial entender a distinção.

O que é o Git?

Pense no Git como um "gestor de histórico" para o seu código. É um **Sistema de Controlo de Versões Distribuído** que instala no seu computador. Ele guarda "fotografias" (chamadas commits) de cada alteração no seu projeto, permitindo-lhe voltar atrás no tempo, ver quem alterou o quê e quando. Acabaram-se os ficheiros como versao_final_final.js.

Em resumo: Git é o software, a ferramenta que você usa localmente para gerir o histórico do seu código.

O que é o GitHub?

O GitHub é uma **plataforma de hospedagem na web** que utiliza o Git. É o local remoto onde guarda os seus projetos (repositórios) para ter um backup na nuvem e, mais importante, para colaborar com outros programadores. O seu perfil no GitHub funciona como um portfólio profissional.

Em resumo: GitHub é o serviço, a plataforma online onde você guarda os seus repositórios Git para partilhar e colaborar.

Pré-requisitos

Antes de começar, garanta que tem o seguinte:

  1. Git Instalado: Se não tiver, pode instalá-lo no seu sistema. No Ubuntu/Debian: 
    sudo apt update && sudo apt install git
  2. Conta no GitHub: Uma conta gratuita em GitHub.com.
  3. Git Configurado: Configure o seu nome de utilizador e email no Git: 
    git config --global user.name "O Seu Nome"
git config --global user.email "o.seu.email@exemplo.com"

Parte 1: Começar um Projeto do Zero (Local)

Passo 1: Criar uma pasta para o projeto

Abra o terminal e crie uma nova pasta para o seu projeto, depois navegue para dentro dela. 

# Cria uma nova pasta chamada 'meu-primeiro-repo'
mkdir meu-primeiro-repo

# Entra na pasta que acabou de criar
cd meu-primeiro-repo

Passo 2: Iniciar o Git na pasta

Este comando "cria" o repositório local, adicionando uma pasta oculta .git para monitorizar as alterações. 

git init

Passo 3: Criar e Adicionar os seus primeiros ficheiros

Vamos criar um ficheiro README.md e adicioná-lo à "staging area" (área de preparação) do Git. 

# Cria um ficheiro README.md com um título dentro
echo "# O Meu Primeiro Repositório" > README.md

# Adiciona o ficheiro à 'staging area'
git add README.md

Passo 4: Fazer o "Commit" das alterações

Um "commit" é um ponto de salvaguarda no histórico do seu projeto. Vamos guardar as alterações que estão na staging area com uma mensagem descritiva. 

git commit -m "Commit inicial: Adiciona o ficheiro README"

Neste ponto, o seu repositório local está criado e já tem um registo no histórico.

Ignorar Ficheiros com .gitignore

Num projeto real, existem muitos ficheiros que não devem ser guardados no histórico do Git. Exemplos incluem dependências (como a pasta node_modules), ficheiros de configuração do seu editor, segredos (chaves de API), ou ficheiros de compilação.

Para dizer ao Git para ignorar estes ficheiros, criamos um ficheiro especial na raiz do projeto chamado .gitignore. 

# Ficheiros de log
*.log

# Dependências do Node.js
node_modules/

# Ficheiros de ambiente com segredos
.env

# Ficheiros do sistema operativo
.DS_Store
Thumbs.db

Cada linha no .gitignore é um padrão de ficheiro ou pasta a ser ignorado. É uma das primeiras coisas que deve criar num novo projeto.

Parte 2: Conectar o seu Projeto Local ao GitHub

Passo 1: Criar um novo repositório no GitHub

  1. Aceda à sua conta no GitHub.com.
  2. No canto superior direito, clique no ícone + e selecione "New repository".
  3. Dê um nome ao repositório (ex: meu-primeiro-repo).
  4. MUITO IMPORTANTE: Deixe as caixas "Add a README file", "Add .gitignore" e "Choose a license" desmarcadas. O repositório deve começar vazio para evitar conflitos.
  5. Clique em "Create repository".

Passo 2: Conectar o seu repositório local ao remoto

O GitHub irá mostrar-lhe uma página com comandos. Iremos usar os da secção "...or push an existing repository from the command line".

1. Adicionar o endereço remoto

Este comando define um "apelido" (origin) para o URL do seu repositório no GitHub. 

# Substitua o URL pelo URL do seu repositório
git remote add origin https://github.com/o-seu-username/meu-primeiro-repo.git

2. Renomear a branch principal para "main"

Esta é uma convenção moderna e uma boa prática. 

git branch -M main

3. Enviar o código (Push) para o GitHub

Este comando envia os seus commits locais para o repositório remoto no GitHub. 

git push -u origin main

Poderá ser pedido para fazer login com as suas credenciais do GitHub no terminal para autorizar esta ação.

Trabalhar no Mesmo Repositório em Vários Computadores

Esta é uma das grandes vantagens de usar o GitHub. O processo para continuar a trabalhar num projeto seu noutro computador é simples e seguro.

O Fluxo de Trabalho

  1. No Computador 1 (onde começou o projeto): Antes de sair, certifique-se de que todas as suas últimas alterações estão guardadas e enviadas para o GitHub. 
    git add .
git commit -m "Últimas alterações do dia"
git push
  2. No Computador 2 (o novo local de trabalho):

    Aqui, em vez de iniciar um novo repositório, você vai "clonar" o que já existe no GitHub.

    1. Abra o terminal e navegue para a pasta onde guarda os seus projetos.
    2. Vá à página do seu repositório no GitHub, clique em <> Code e copie o URL HTTPS.
    3. Execute o comando git clone: 
      git clone https://github.com/o-seu-username/meu-primeiro-repo.git

      Isto irá descarregar o projeto, todo o seu histórico, e configurar automaticamente a ligação ao GitHub.

  3. Trabalhar no Computador 2: Faça as suas alterações, commits e, quando terminar, envie tudo de volta para o GitHub com git push.
  4. De volta ao Computador 1: Este é o passo crucial. Antes de voltar a trabalhar no computador original, tem de "puxar" as alterações que fez no Computador 2. 
    # Sincroniza o seu repositório local com as últimas alterações do GitHub
git pull

    Manter este ciclo de push (ao sair de um PC) e pull (ao chegar a outro) garante que ambos os locais estão sempre sincronizados.

Trabalhar com Branches (Ramos)

Branches são a superpotência do Git. Pense no branch principal (main) como o tronco estável de uma árvore. Para qualquer nova funcionalidade, correção de bug ou experimentação, você cria um novo branch (um ramo) para trabalhar em total isolamento, sem afetar a versão principal do projeto.

O Ciclo de Vida de um Branch

  1. Criar um novo Branch: Crie um branch com um nome descritivo a partir do main. 
    # Cria um novo branch e muda para ele num só passo
git checkout -b adicionar-pagina-contacto
  2. Trabalhar e Fazer Commits: Faça todas as alterações e commits que precisar. O seu trabalho está seguro e isolado neste branch. O main permanece limpo. 
    git add .
git commit -m "Cria a página de contacto com formulário"
  3. Publicar o Branch (se necessário): Se precisar que outros vejam o seu trabalho ou se quiser abrir um Pull Request, tem de enviar o seu novo branch para o GitHub. 
    # Envia o branch para o repositório remoto (origin) e cria uma ligação
git push --set-upstream origin adicionar-pagina-contacto

    Dica: Da próxima vez que fizer um push a partir deste branch, só precisa de git push.

  4. Juntar de volta ao Principal (Merge): Quando o trabalho estiver concluído e testado, volte ao branch principal e junte as alterações do seu branch. 
    git checkout main
git merge adicionar-pagina-contacto
  5. Limpeza Local: Após o merge, o branch de funcionalidade já não é necessário localmente e pode ser apagado para manter o seu repositório organizado. 
    # Apaga o branch localmente
git branch -d adicionar-pagina-contacto
  6. Limpeza Remota: Apagar o branch localmente não o remove do GitHub. É uma boa prática também o apagar do repositório remoto para evitar acumular branches antigos. 
    # Apaga o branch no repositório remoto (origin)
git push origin --delete adicionar-pagina-contacto

Guardar Alterações Temporariamente (git stash)

Imagine que está a meio de uma alteração complexa, mas precisa de mudar urgentemente de branch para corrigir um bug. Não pode fazer commit de código inacabado. É aqui que git stash se torna um "salva-vidas".

O stash guarda as suas modificações numa "prateleira" temporária, deixando a sua área de trabalho limpa para poder mudar de branch.

  1. Guardar as alterações: 
    # Guarda as alterações não commitadas na "prateleira"
git stash
  2. Mudar de branch e trabalhar: Agora a sua área está limpa. Pode fazer git checkout outro-branch, corrigir o bug, e voltar.
  3. Recuperar as alterações: Quando voltar ao seu branch original, pode recuperar o que guardou. 
    # Aplica o último stash e remove-o da lista
git stash pop

Ver Branches Existentes

Ver os branches existentes é uma tarefa comum para ter uma visão geral do projeto e saber em que a equipa está a trabalhar.

1. Ver Branches Locais

Mostra os branches que existem no seu computador. O branch atual é marcado com *. 

git branch

2. Ver Branches Remotos

Mostra os branches que existem no repositório remoto (ex: GitHub). 

git branch -r

3. Ver Todos os Branches (Locais e Remotos)

A opção mais completa, que combina as duas listas anteriores. 

git branch -a

Colaboração: Fork, Clone e Pull Requests

Este é o fluxo de trabalho essencial para contribuir para projetos em equipa ou open-source, especialmente quando não tem permissão para escrever diretamente no repositório principal.

Passo 1: Fork – A Sua Cópia Pessoal no GitHub

Um fork é uma cópia de um repositório que fica na sua própria conta do GitHub. Pense nele como a sua "sandbox" pessoal. Você pode fazer as alterações que quiser no seu fork sem afetar o projeto original. Para o criar, basta clicar no botão "Fork" na página do repositório original.

Passo 2: Clone – A Cópia no Seu Computador

Depois de ter o seu fork, você clona (git clone) essa sua cópia pessoal para o seu computador para poder trabalhar nos ficheiros. 

git clone https://github.com/o-seu-username/nome-do-repo-forked.git

Passo 3: Pull Request (PR) – Propor as Suas Alterações

Depois de fazer as alterações localmente (num novo branch, claro!), você envia-as para o seu fork no GitHub. A partir daí, o GitHub permite-lhe abrir um Pull Request. Este é um pedido formal ao dono do projeto original para que ele "puxe" (pull) e junte (merge) as suas alterações ao projeto principal. Um bom PR tem um título claro e uma descrição detalhada do que foi feito e porquê, iniciando um processo de revisão e discussão.

Resolver Conflitos de Merge

Um conflito acontece quando o Git tenta juntar dois branches que alteraram as mesmas linhas do mesmo ficheiro de formas diferentes. O Git não consegue adivinhar qual é a versão correta, por isso pára e pede a sua ajuda.

Como Identificar e Resolver

Ao abrir o ficheiro em conflito, o Git insere marcadores visuais para lhe mostrar as duas versões: 

<<<<<<< HEAD // Início da versão do seu branch atual (para onde está a fazer merge)
<h1>Bem-vindo ao Portfolio Profissional</h1>
======= // Separa as duas versões em conflito
<h1>O Fantástico Portfolio de Pedro</h1>
>>>>>>> feature/mudar-titulo // Fim da versão do branch que está a tentar juntar

A sua tarefa é ser o editor: apague os marcadores (<<<, ===, >>>) e edite o código para que fique com a versão final que deseja (pode manter uma, outra ou combinar as duas). Depois de guardar o ficheiro, finalize o processo: 

# Diga ao Git que o ficheiro foi resolvido
git add nome-do-ficheiro-resolvido.html

# Crie um commit para concluir o merge
git commit

Avançado: Sincronizar com `git rebase`

O git rebase é uma alternativa ao git merge para integrar alterações de um branch noutro. A principal diferença é que o rebase reescreve a história dos commits para criar um histórico linear e mais limpo.

Imagine que, enquanto trabalhava no seu branch, o branch main recebeu novas atualizações. Em vez de criar um "commit de merge" que junta as duas histórias, o rebase move o seu branch para que ele comece no final do main atualizado, aplicando os seus commits um a um. 

# No seu branch de funcionalidade, para sincronizar com as últimas alterações do main
git fetch origin main
git rebase origin/main
AVISO: Nunca faça rebase em branches públicos/partilhados que outras pessoas possam estar a usar. Como ele reescreve a história, pode causar grandes problemas para os seus colegas. Use-o principalmente para limpar o seu histórico em branches locais antes de o partilhar.

Ver o Histórico de Commits (git log)

Esta é uma das operações mais úteis no Git. O comando git log tem várias opções para formatar e filtrar a saída.

Formatos Populares 

# Ver cada commit numa única linha (muito útil)
git log --oneline

# Ver um gráfico de branches
git log --graph --oneline

# Ver estatísticas dos ficheiros alterados
git log --stat

Filtrar a Lista 

# Ver os últimos 3 commits
git log -n 3

# Ver commits de um autor específico
git log --author="Pedro Santos"

# Ver o histórico de alterações de um ficheiro
git log -- src/database.js

Voltar a um Commit Anterior (CLI)

É crucial entender as diferentes formas de o fazer. A escolha depende do seu objetivo.

1. Apenas para Inspeção: git checkout

Para "viajar no tempo" e ver como o projeto estava num ponto específico, sem o risco de perder trabalho. Para voltar ao presente, use git checkout main. 

git checkout 3b4c1a9

2. Reverter um Commit Público: git revert

Se o commit já foi partilhado (git push), esta é a forma mais segura. Em vez de apagar, cria um novo commit que desfaz as alterações, preservando o histórico. 

git revert f3b2c1a

3. Desfazer Commits Locais: git reset

git reset move o ponteiro do seu branch, reescrevendo o histórico. Use com muito cuidado e apenas em commits que ainda não partilhou.

AVISO: A opção --hard apaga permanentemente todas as alterações. Use apenas se tiver a certeza absoluta de que quer descartar o trabalho. 
# Apaga permanentemente o trabalho feito após o commit f3b2c1a
git reset --hard f3b2c1a

Alternativa: Usando a Interface do GitHub

A interface do site é ótima para visualizar o histórico e fazer reversões de forma segura, sem precisar de memorizar comandos.

Visualizar o Histórico no Site

  1. Na página do seu repositório, clique no link que diz "commits" (acima da lista de ficheiros).
  2. Pode clicar em cada commit para ver as alterações exatas.

Reverter um Commit no Site

Esta ação faz o equivalente a um git revert, preservando o histórico.

  1. Navegue até à lista de commits e clique no que deseja reverter.
  2. Na página do commit, clique no botão com três pontinhos (...) e selecione "Revert".
  3. O GitHub irá guiá-lo para criar um Pull Request (PR) com a reversão.
  4. Faça o merge do Pull Request para aplicar a alteração no branch principal.

Gestão de Projeto no GitHub

O GitHub é mais do que um local para guardar código. É uma plataforma completa de gestão de projetos.

Tags e Releases

Uma "tag" é um marcador que aponta para um commit específico, usado normalmente para marcar pontos importantes como versões de lançamento (v1.0, v1.1). 

# Cria uma tag chamada v1.0 no commit atual
git tag -a v1.0 -m "Versão de Lançamento 1.0"

# Envia todas as suas tags para o GitHub
git push --tags

No GitHub, pode ir à secção "Releases" e criar uma nova "release" a partir de uma tag, adicionando notas de lançamento e ficheiros compilados (executáveis, por exemplo).

GitHub Issues

As "Issues" são o centro de discussão e planeamento de um projeto. Use-as para:

  • Reportar bugs.
  • Sugerir novas funcionalidades.
  • Fazer perguntas e discutir ideias.
  • Organizar tarefas e atribuí-las a membros da equipa.

Pode ligar commits e Pull Requests a issues, criando um fluxo de trabalho rastreável (ex: "Este commit corrige o bug #42").

Bónus: Publicar com GitHub Pages

O GitHub Pages é um serviço gratuito que permite hospedar um site simples diretamente a partir do seu repositório. É perfeito para portfólios, páginas de projetos ou documentação.

Pré-requisito: Para que funcione, o seu repositório deve ter um ficheiro chamado index.html na raiz do projeto.

Passo 1: Ir às Definições

Na página do seu repositório no GitHub, clique na aba "Settings".

Passo 2: Secção "Pages"

No menu lateral esquerdo, clique em "Pages".

Passo 3: Configurar a Fonte

Na secção "Build and deployment", em "Source", selecione "Deploy from a branch".

Depois, em "Branch", selecione a branch main e a pasta /(root). Clique em "Save".

Passo 4: Visitar a Página

O GitHub irá demorar um ou dois minutos a publicar o seu site. Após esse tempo, a página irá recarregar e mostrar-lhe um link com o endereço da sua nova página, algo como: 

https://o-seu-username.github.io/nome-do-repo/

Parabéns! O seu projeto está agora online.

Resumo do Fluxo de Trabalho Diário

Após a configuração inicial, o seu processo diário será:

  1. Sincronize com o remoto (importante em equipa): 
    git pull
  2. Trabalhe nos seus ficheiros (crie, edite, apague).
  3. Adicione as alterações à staging area: 
    git add .
  4. Faça o commit das alterações com uma mensagem: 
    git commit -m "Mensagem descritiva do que fiz"
  5. Envie as alterações para o GitHub: 
    git push
